# Delaem Digital Visual Direction Lock v1

## 1. Visual Concept Name

**Founder Control Room / ClientFlow Diagnostic Interface**

Версия: v1  
Проект: «Делаем Диджитал» / ClientFlow System  
Назначение: зафиксировать визуальное направление перед implementation layer.

## 2. Main Metaphor

Виталий Тимошенко - архитектор маршрута привлечения клиентов.

Сайт - digital control room, где сайт, квиз, трафик, заявки, follow-up и аналитика собраны в одну управляемую систему.

Метафора должна считываться через:

- founder scene;
- route-lines;
- diagnostic interface;
- ClientFlow path;
- системную инфографику;
- премиальные темные surfaces;
- минимальный, но точный визуальный язык.

Сайт не должен выглядеть как:

- generic SaaS dashboard;
- cyberpunk interface;
- AI-шум;
- лендинг с набором темных карточек;
- портфолио без маршрута.

## 3. Hero Direction

Hero должен быть флагманской founder-сценой.

### Смысл hero

За 3 секунды пользователь должен понять:

- кто Виталий;
- что он CEO «Делаем Диджитал»;
- что он архитектор ClientFlow System;
- что есть три направления: Сайт / Трафик / Система;
- куда нажать для следующего шага.

### Композиция

Не делать:

- просто портрет справа;
- обычную split-layout секцию;
- декоративную фотографию без связи с системой;
- тяжелую audit-card поверх hero.

Нужно:

- интегрировать портрет в system scene;
- вокруг портрета показать subtle diagnostic interface;
- использовать route-line как визуальное продолжение ClientFlow;
- добавить ощущение control-room глубины;
- сохранить CTA читаемым;
- сохранить текущий headline и смысл.

### Роль портрета

Портрет Виталия должен показывать:

- личную ответственность;
- экспертность;
- статус founder;
- роль архитектора маршрута.

Портрет не должен:

- доминировать над смыслом;
- быть просто декоративной фотографией;
- выглядеть как stock founder shot;
- ломать mobile first screen.

### Hero Visual Elements

Разрешено:

- subtle grid;
- route-lines;
- edge highlights;
- small diagnostic nodes;
- ClientFlow path fragments;
- restrained copper accents;
- mineral steel highlights;
- depth behind portrait.

Запрещено:

- blue SaaS glow;
- purple gradients;
- random neon;
- cyberpunk;
- noisy dashboard widgets;
- meaningless metrics.

## 4. ClientFlow Visual

Главная формула:

**Оффер -> Сайт/Квиз -> Трафик -> Заявка -> Follow-up -> Аналитика**

Эта формула должна стать центральной инфографикой сайта.

### Как показывать

Нужно:

- показать последовательность как маршрут;
- связать шаги визуальной линией;
- дать каждому шагу роль;
- показать, что система управляемая;
- сделать формулу запоминаемой.

Не нужно:

- делать шесть одинаковых карточек ради карточек;
- добавлять графики без данных;
- превращать в перегруженный dashboard;
- делать route-line как чистый декор.

### Visual Behavior

- desktop: маршрут может быть горизонтальным, диагональным или control-room panel;
- mobile: маршрут должен складываться в ясную вертикальную последовательность;
- каждый шаг должен читаться без зума;
- линия должна объяснять связь, а не украшать блок.

## 5. Product Cards

Продукты:

1. Делаем Сайт - от 150 000 ₽
2. Делаем Трафик - от 250 000 ₽
3. Делаем Систему - от 350 000 ₽

### Product Line Role

Три продукта должны выглядеть как продуктовая линейка, а не как случайные pricing cards.

Нужно показать:

- входной продукт;
- лидогенерационный продукт;
- флагманский продукт;
- разницу между ними;
- почему «Делаем Систему» главный.

### Featured Product

«Делаем Систему» должен быть визуально главным:

- сильнее surface;
- заметнее depth;
- более уверенный badge;
- более высокий visual priority;
- CTA не должен теряться.

Но нельзя:

- менять цену;
- менять смысл;
- делать агрессивный sales pressure;
- добавлять fake scarcity;
- добавлять неподтвержденные бонусы.

### Card Style

Карточки не должны быть generic SaaS cards.

Нужно:

- меньше шаблонности;
- больше архитектурности;
- clear hierarchy;
- premium borders;
- restrained highlights;
- читаемые benefits;
- аккуратные CTA states.

## 6. Palette

Основное направление:

- graphite / black base;
- restrained copper / bronze accents;
- mineral steel highlights;
- warm off-white text;
- muted gray secondary text.

### Palette Roles

| Роль | Направление |
|---|---|
| Background | deep graphite / near black |
| Primary surface | black graphite with subtle depth |
| Secondary surface | dark mineral gray |
| Primary text | warm off-white |
| Muted text | controlled gray / stone |
| Accent | restrained copper / bronze |
| Highlight | mineral steel, not blue SaaS |
| Border | low-opacity warm metal |

### Forbidden Palette

Запрещено:

- blue SaaS glow;
- purple gradients;
- cyberpunk neon;
- dominant beige/brown theme;
- random multicolor accents;
- one-note palette без контраста.

## 7. Typography

Fonts:

- Headings: Manrope.
- Body: Onest.

Правила:

- strong hierarchy;
- hero headline крупный, но не кричащий;
- section titles плотные и уверенные;
- body text короткий и читаемый;
- UI labels не делать микроскопическими;
- no text walls;
- no decorative letter spacing abuse;
- не масштабировать font-size через viewport width без контроля.

Typography должна поддерживать ощущение:

- founder authority;
- system clarity;
- premium service;
- readable diagnostic interface.

## 8. UI Rules

Общее направление:

- premium dark;
- strong spacing;
- fewer but stronger surfaces;
- intentional cards;
- expensive clear buttons;
- premium request panels;
- route-lines только там, где объясняют system logic.

### Surfaces

Surfaces должны:

- выглядеть как части одной системы;
- иметь понятную иерархию;
- не превращаться в бесконечную сетку карточек;
- использовать depth умеренно.

### Buttons

Buttons должны:

- быть ясными;
- иметь дорогой tactile feel;
- не выглядеть как стандартные SaaS buttons;
- сохранять хороший contrast;
- иметь аккуратный hover/focus.

Primary CTA:

- визуально главный;
- теплый copper/bronze direction;
- без neon glow.

Secondary CTA:

- спокойный;
- не конкурирует с primary;
- ясно читается.

### Cards

Cards должны:

- иметь функцию;
- усиливать hierarchy;
- быть разными по роли;
- не выглядеть как повтор одного шаблона.

### Forms

Форма должна выглядеть как premium request panel:

- ясно показывает следующий шаг;
- не выглядит как техническая анкета;
- поля читаются;
- consent виден;
- legal links сохранены;
- backend behavior не меняется.

## 9. Motion

Motion level: subtle.

Разрешено:

- route-line reveal;
- soft fade;
- button microinteractions;
- gentle hover depth;
- small diagnostic node accent.

Запрещено:

- aggressive parallax;
- glow pulse;
- heavy intro animation;
- infinite distracting loops;
- motion, который мешает чтению;
- animation without purpose.

Reduced motion:

- все декоративные motion effects должны иметь безопасное поведение при `prefers-reduced-motion`.

## 10. Mobile Rules

Mobile должен быть самостоятельным опытом, а не сжатым desktop.

Правила:

- first screen remains clear;
- CTA visible early;
- portrait must not dominate uselessly;
- product cards readable;
- cookie must not fight with hero;
- no horizontal scroll;
- route formula reads vertically;
- form fields are easy to tap;
- text does not overflow;
- no tiny unreadable labels.

Hero на mobile:

- headline и CTA важнее portrait;
- portrait можно сделать ниже, компактнее или интегрировать мягче;
- diagnostic effects должны быть сильно упрощены.

Products на mobile:

- «Делаем Систему» остается главным;
- карточки читаются без зума;
- CTA не уезжает слишком далеко.

## 11. Forbidden

Запрещенные направления:

- generic SaaS dashboard;
- random glowing cards;
- cyberpunk;
- blue/purple gradients;
- noisy charts;
- fake metrics;
- overdesigned AI clutter;
- long copy;
- «гарантируем заявки»;
- fake cases;
- fake awards;
- декоративные графики без смысла;
- тяжелая audit-card поверх hero;
- портрет как отдельная случайная фотография;
- бессмысленные dashboard widgets.

## 12. Acceptance Criteria for Future Visual Build

Будущий implementation layer считается успешным, если:

- hero looks like flagship founder site;
- Виталий визуально встроен в ClientFlow-систему;
- за 3 секунды понятно: кто, что делает, какие продукты, куда нажать;
- ClientFlow route is visually memorable;
- формула стала центральной инфографикой, а не набором карточек;
- product ladder is instantly clear;
- «Делаем Систему» визуально главный продукт;
- final CTA feels premium;
- форма выглядит как request panel;
- site no longer looks like generic dark landing;
- no meaning changes;
- no backend changes;
- no legal changes;
- no nav/price/product changes;
- mobile remains clear and usable.

## 13. Implementation Notes for Next Layer

Следующий implementation spec должен:

- ссылаться на этот visual direction lock;
- сначала определить секции и файлы;
- не начинать с хаотичного CSS tuning;
- запрещать изменение runtime behavior;
- явно защищать форму, legal, prices, nav labels, English page;
- включать desktop/mobile QA;
- включать build/typecheck;
- при необходимости включать preview deploy.
