# Delaem Digital Reference Pack v1

## 1. Purpose

Этот reference pack нужен для следующего visual implementation layer сайта «Делаем Диджитал».

Задача документа — убрать случайный дизайн и зафиксировать, какие визуальные паттерны брать из утвержденных hero-референсов пользователя. Это не подборка для буквального копирования, а система извлечения приемов: композиция, масштаб, роль портрета, типографика, CTA, глубина сцены и уровень арт-дирекшна.

Будущий visual layer должен опираться на этот документ вместе с:

- `docs/website-factory/delaem-digital-project-brief-v1.md`
- `docs/website-factory/delaem-digital-visual-direction-lock-v1.md`
- `docs/website-factory/delaem-digital-implementation-spec-v1.md`
- `docs/website-factory/reference-scoring-system-v1.md`

## Master Hero Direction

- master file: `Hero/hero_master_direction_v1.jpg`;
- role: primary visual direction for future v0.15.0 implementation;
- use as main art-direction anchor;
- supporting references must not override it.

Что берем из master reference:

- dark graphite founder scene;
- portrait as authority anchor;
- strict premium typography;
- restrained copper / bronze accent;
- calm control-room mood;
- diagnostic panel below / near hero;
- minimal copy;
- high trust, no hype;
- business / system architecture feeling.

Что нельзя брать из supporting references, если это конфликтует с master:

- blue SaaS glow;
- purple jellyfish;
- acid red;
- cyberpunk / NFT;
- oversized random 3D objects;
- aggressive event / speaker landing style;
- fake metrics;
- visual noise.

## 2. What User Actually Likes

В присланных hero-референсах пользователю нравится не конкретная сетка или эффект, а общее ощущение сильного первого экрана:

- cinematic founder / agency hero;
- крупный visual center;
- смелая типографика;
- минимум текста;
- темная или глубокая премиальная сцена;
- портрет как часть композиции, а не отдельная фотография;
- strong CTA, заметный, но не кричащий;
- премиальные контрастные surfaces;
- ощущение арт-дирекшна;
- сайт как брендовый постер + конверсионная система.

Ключевой вывод: сайт должен ощущаться не как набор секций, а как режиссированная digital-сцена, где бренд, основатель и продуктовая логика видны за первые секунды.

## 3. What To Avoid

Запрещенные направления для будущего visual implementation:

- generic dark SaaS landing;
- мелкие карточки вместо композиции;
- random route-lines;
- dashboard clutter;
- blue SaaS glow;
- purple jellyfish style;
- cyberpunk / NFT;
- acidic red;
- fake metrics;
- stock founder shot;
- портрет просто справа;
- дизайн в логике «чуть больше свечения и линий».

Если визуальное решение можно описать как «добавили сетку, линии и glow поверх текущих блоков», оно не соответствует reference pack.

## 4. Hero Pattern Extraction

### A. Cinematic Founder Scene

Что брать:

- крупный портрет или объект как главный visual center;
- сильная атмосфера первого экрана;
- свет, глубина и контраст как часть композиции;
- текст встраивается в сцену, а не стоит отдельно;
- CTA остается видимым и понятным.

Как применять:

- портрет Виталия должен стать центром постановки;
- вокруг портрета должна быть среда ClientFlow, а не набор случайных карточек;
- текст должен поддерживать сцену и не дробить первый экран.

Риск:

- можно уйти в красивый постер без ясного оффера.

### B. Poster Typography Hero

Что брать:

- большая типографика;
- мало слов;
- visual identity через масштаб;
- сильный контраст между текстом, портретом и фоном;
- ощущение брендового манифеста.

Как применять:

- headline должен быть крупным и уверенным;
- secondary copy должен быть коротким;
- CTA должен находиться рядом с основным смыслом.

Риск:

- можно потерять ясность, если типографика станет важнее оффера.

### C. Founder + Product Interface

Что брать:

- портрет + интерфейсные фрагменты;
- интерфейс как смысловой слой, а не декор;
- ощущение, что основатель управляет системой;
- подходит под ClientFlow, потому что метод связан с маршрутом и диагностикой.

Как применять:

- вокруг Виталия можно показывать элементы маршрута: Сайт, Трафик, Заявка, Follow-up, Аналитика;
- интерфейсные фрагменты должны объяснять ClientFlow, а не изображать fake dashboard;
- floating UI допустим только при смысловой роли.

Риск:

- нельзя перегрузить hero dashboard-виджетами.

### D. Premium Agency Hero

Что брать:

- брендовый первый экран;
- уверенный CTA;
- минимум объяснений;
- плотная, дорогая композиция;
- ощущение студии или founder-led agency с характером.

Как применять:

- «Делаем Диджитал» должен ощущаться как сильный бренд, а не технический подрядчик;
- первый экран должен выглядеть как вход в премиальный сервис;
- CTA должен быть спокойным и статусным.

Риск:

- не стать просто «креативной студией» без продуктовой логики ClientFlow.

### E. Route / System Hero

Что брать:

- маршрут как визуальная метафора;
- понятная связь между этапами;
- ощущение управляемой системы;
- визуальная поддержка формулы ClientFlow.

Как применять:

- маршрут должен быть вторым visual anchor после портрета;
- формула должна читаться как движение: Сайт → Трафик → Заявка → Follow-up → Аналитика;
- линии и узлы нужны только там, где объясняют связь.

Риск:

- нельзя превращать hero в схему из карточек.

## 5. Best Direction For Delaem Digital

Основное направление:

**Hybrid: Cinematic Founder Control Room + ClientFlow Route Interface.**

Суть:

Виталий — главный visual anchor. ClientFlow route — второй visual anchor.

Hero должен выглядеть как премиальная founder-сцена, где вокруг Виталия собран маршрут:

**Сайт → Трафик → Заявка → Follow-up → Аналитика**

Это не SaaS-dashboard и не обычный split layout. Это постановка: основатель как архитектор маршрута привлечения клиентов, а сайт как control room, где видны ключевые точки системы.

Иерархия первого экрана:

1. Виталий как центр доверия и автор метода.
2. Большой headline как брендовая позиция.
3. ClientFlow route как визуальное доказательство системности.
4. CTA как понятный следующий шаг.

## 6. Translation To Current Website

### Hero

Нужно:

- сделать портрет крупнее и сценичнее;
- отказаться от ощущения «портрет в правой колонке»;
- построить фон как cinematic control room;
- использовать минимум floating panels;
- усилить крупную типографику;
- сохранить ясность оффера;
- показать роль Виталия как архитектора ClientFlow System.

Нельзя:

- возвращаться к обычному SaaS split;
- перегружать hero виджетами;
- делать route-lines без смысла;
- прятать CTA за декоративной сценой.

### ClientFlow

Нужно:

- сделать route-инфографику, не карточки;
- путь должен быть главным визуальным элементом секции;
- формула должна читаться как движение по системе;
- desktop может использовать широкий маршрут;
- mobile должен переводить маршрут в чистую вертикальную последовательность.

Нельзя:

- делать шесть одинаковых карточек;
- добавлять fake charts;
- превращать секцию в dashboard.

### Products

Нужно:

- продуктовая линейка должна выглядеть как premium offer cards;
- «Делаем Систему» — flagship;
- иерархия трех продуктов должна быть понятна сразу;
- CTA должен быть заметным, но без давления.

Нельзя:

- менять цены;
- менять названия продуктов;
- представлять «Делаем Трафик» как полноценный сайт;
- добавлять scarcity, fake badges или fake metrics.

### Audit

Нужно:

- сделать audit diagnostic entry point;
- показать, что это разбор маршрута: сайт / трафик / система;
- держать блок компактным и статусным.

Нельзя:

- добавлять fake scores;
- делать тяжелый dashboard;
- превращать audit в длинное объяснение.

### Final CTA

Нужно:

- сделать premium request panel;
- форма должна ощущаться как логичный запрос на разбор;
- сохранить поля, validation, consent, backend и Telegram behavior.

Нельзя:

- менять API;
- менять legal consent;
- добавлять лишние поля;
- делать форму обычным серым блоком внизу страницы.

## 7. Reference Scoring Matrix

Каждый будущий референс оценивается по шкале 0-100.

| Критерий | Вес | Что оцениваем |
| --- | ---: | --- |
| Cinematic strength | 20 | Насколько первый экран ощущается как сильная сцена, а не набор блоков |
| Founder / portrait integration | 15 | Насколько портрет или объект встроен в композицию |
| Typography impact | 15 | Насколько типографика создает масштаб и характер |
| Conversion clarity | 15 | Насколько быстро понятно, куда нажать и зачем |
| ClientFlow / system adaptability | 15 | Можно ли адаптировать паттерн под маршрут ClientFlow |
| Premium feel | 10 | Насколько сцена выглядит дорого и уверенно |
| Mobile adaptability | 5 | Можно ли сохранить силу композиции на mobile |
| Avoids forbidden styles | 5 | Нет ли blue glow, cyberpunk, dashboard clutter, fake metrics |

Интерпретация:

- 85-100: сильный референс, можно брать как основной паттерн;
- 70-84: полезный референс, брать отдельные приемы;
- 50-69: использовать осторожно, только один аспект;
- 0-49: не использовать для текущего сайта.

## 8. Implementation Rules For Future v0.15.0

Будущий Codex visual layer обязан:

- сначала прочитать этот reference pack;
- first inspect `Hero/hero_master_direction_v1.jpg`;
- use it as primary visual anchor;
- then inspect 19 supporting references only for secondary patterns;
- do not average all references into mixed style;
- открыть reference screenshots из `Hero/`;
- сравнить текущий hero с этими референсами;
- извлечь композицию и паттерны до правок CSS / JSX;
- не реализовывать дизайн только по текстовому описанию;
- не делать CSS-косметику;
- не добавлять случайные линии;
- не делать обычный dark SaaS;
- строить hero как сцену;
- делать ClientFlow как маршрут;
- сохранять продукты, цены, форму, backend;
- не менять смысл;
- не добавлять fake metrics;
- не добавлять fake cases;
- не добавлять fake awards;
- не писать «гарантируем заявки».

Before v0.15.0 visual implementation:

- first inspect `Hero/hero_master_direction_v1.jpg`;
- use it as primary visual anchor;
- then inspect 19 supporting references only for secondary patterns;
- do not average all references into mixed style;
- open `Hero/` reference screenshots;
- compare current hero against these references;
- extract composition / patterns before touching CSS / JSX;
- do not implement from words only.

Минимальная логика реализации:

1. Сначала построить hero-композицию.
2. Затем перевести ClientFlow в маршрут.
3. Затем подчинить Products, Audit и Final CTA этой же visual system.
4. Только после этого полировать surfaces, buttons, borders и microinteractions.

Если hero не выглядит как premium founder scene, слой считается неготовым независимо от качества остальных секций.
