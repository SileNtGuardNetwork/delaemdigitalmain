# Rejected Visual Patterns v1

## Purpose

This document records visual directions that have already been tested or explicitly rejected for Delaem Digital. Its job is to prevent future visual passes from reintroducing patterns that made the site feel generic, heavy, dashboard-like, or less premium.

Rejected patterns must not be reintroduced unless the user explicitly approves that specific pattern again.

## D3.2 Hero Rejections

### Route-Card Grid Under Hero CTA

**Rejected because:** it turned the hero into a stacked interface block and made the first viewport feel like a dark SaaS dashboard rather than a cinematic founder-led scene.

**Do instead:** communicate ClientFlow through thin route lines, subtle labels, atmospheric sequence cues, or architectural linework around the founder scene.

### Operator Panel Over Portrait

**Rejected because:** it made the founder image feel like a dashboard container and competed with the human authority anchor.

**Do instead:** integrate the portrait with subtle masks, rim light, edge lines, and quiet route cues that do not cover the face or body.

### Dashboard-Like Hero Widgets

**Rejected because:** widgets suggested fake software UI and pulled the hero toward generic SaaS instead of high-ticket founder authority.

**Do instead:** use atmosphere, composition, lighting, and minimal labels. If a system signal appears, it must map to real ClientFlow logic and stay visually secondary.

### Fake Command Center UI

**Rejected because:** literal command-center boxes made the visual concept too obvious and artificial.

**Do instead:** make the command-center feeling cinematic and architectural: controlled grid, signal lines, dark depth, and founder-as-operator presence.

### Mobile Tower

Rejected sequence:

> offer -> CTA -> route panel -> operator panel -> portrait

**Rejected because:** it delayed the founder anchor, created a long stack of dark boxes, and made mobile feel functional rather than premium.

**Do instead:** keep H1 and CTA early, bring founder presence sooner, and compress system logic into one small line or ambient cue.

### Photo-Card Framing

**Rejected because:** a rectangular portrait card felt pasted beside the copy instead of integrated into one scene.

**Do instead:** use softer edge treatment, masking, atmospheric shadows, subtle linework, and scene-level lighting so the portrait belongs to the composition.

### Too Many Equal-Weight Dark Cards

**Rejected because:** equal visual weight destroys hierarchy and makes the page feel like a template.

**Do instead:** use fewer, stronger elements. Cards should be reserved for meaningful repeated content, not every concept or signal.

## Operating Rule

Before any visual implementation, check this file. If a proposed change resembles a rejected pattern, choose a quieter, more architectural alternative or ask for explicit approval.
