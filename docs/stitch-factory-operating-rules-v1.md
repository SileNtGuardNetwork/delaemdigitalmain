# Stitch Factory Operating Rules v1

## Назначение

Этот документ фиксирует, как использовать Stitch в фабрике Delaem Digital. Stitch нужен для визуального поиска и проверки направлений, а не для команды "сделай весь сайт".

Stitch должен получать уже подготовленный контекст: нишу, оффер, структуру, фото, референсы, анти-паттерны и ограничения. Результат Stitch должен затем переводиться в Next.js через Claude Code или Codex по утвержденному `DESIGN.md`.

## Правильный Порядок Работы

1. Сначала создать или адаптировать `DESIGN.md`.
2. Затем получить 2-3 визуальных направления.
3. Затем выбрать направление и сделать 3 варианта hero.
4. Затем собрать ключевые экраны и состояния.
5. Затем утвердить дизайн.
6. Затем передать дизайн в Claude Code / Codex для Next.js-реализации.

Нельзя начинать с промпта "сделай весь сайт". Это почти всегда приводит к шаблонному SaaS, слабой структуре, случайным блокам и плохой мобильной компрессии.

## Входные Данные Для Stitch

Перед запуском Stitch должны быть переданы:

- ниша клиента;
- целевая аудитория;
- основной оффер;
- целевое действие;
- структура страницы или sitemap;
- `DESIGN.md`;
- доступные фото и их роль;
- обязательные CTA и ссылки;
- референсы: что взять и что не копировать;
- анти-паттерны из `docs/rejected-visual-patterns-v1.md`;
- правила бренда и токены;
- требования по русскому интерфейсу;
- ограничения по backend, формам, аналитике и legal-блокам.

## DESIGN.md Сначала

`DESIGN.md` должен описывать:

- бизнес-контекст;
- визуальную гипотезу;
- структуру страницы;
- роль каждого экрана;
- hero-направление;
- фото-стратегию;
- палитру и типографику;
- CTA-логику;
- мобильное поведение;
- запреты;
- критерии приемки.

Stitch не должен угадывать стратегию. Stitch должен визуализировать уже выбранную стратегию.

## 2-3 Визуальных Направления

На первом визуальном проходе Stitch должен дать 2-3 разных направления, а не один случайный макет.

Каждое направление должно включать:

- короткое описание идеи;
- desktop hero;
- mobile hero;
- подход к фото;
- подход к CTA;
- чем направление отличается от остальных;
- риски: где может стать generic, dashboard-like или шаблонным.

Для Delaem Digital предпочтительные направления должны оставаться founder-led, темными, премиальными, архитектурными и конверсионными.

## 3 Hero Варианта

После выбора направления Stitch должен сделать 3 hero-варианта.

Варианты должны отличаться не цветом, а композицией:

- вариант с сильным портретом;
- вариант с более явной ClientFlow-архитектурой;
- вариант с editorial/control-room балансом.

Hero должен проверяться отдельно, потому что слабый hero ломает весь сайт.

Минимум для hero:

- оффер понятен за 5 секунд;
- CTA виден в первом экране;
- портрет интегрирован в сцену;
- нет ощущения `текст слева + фото-карточка справа`;
- нет фейкового dashboard;
- нет prompt bar;
- нет chip cloud.

## Ключевые Экраны

После hero Stitch должен собрать ключевые экраны:

- первый экран desktop;
- первый экран mobile;
- problem / diagnostic;
- ClientFlow / process;
- offer / pricing;
- lead form / quiz;
- final CTA;
- footer / legal support.

Не нужно требовать от Stitch полный production-сайт на первом проходе. Сначала проверяются главные конверсионные и визуальные решения.

## Русский Интерфейс

Все видимые UI-тексты в макете должны быть на русском языке.

Запрещено оставлять:

- английские CTA;
- английские labels в бизнес-facing интерфейсе;
- lorem ipsum;
- generic SaaS phrases;
- технические подписи, которые посетитель не должен видеть.

Исключение: внутренние имена файлов, технические комментарии в передаваемых артефактах и системные названия инструментов.

## Запрещенные Паттерны

Stitch-промпты должны явно запрещать:

- generic SaaS landing;
- dashboard hero;
- фейковые метрики;
- prompt/search bar hero;
- chip cloud;
- purple AI style;
- violet glow identity;
- crypto/cyber визуал;
- photo-card hero;
- split `text-left + image-right`;
- бесконечные темные карточки;
- стеклянные панели в каждом блоке;
- декоративные графики без смысла;
- product UI, если реального продукта или данных нет.

Если Stitch вернул один из этих паттернов, результат считается не направлением, а черновиком для отклонения.

## Что Передавать В Stitch-Промпт

Минимальный промпт должен содержать:

```text
Создай визуальное направление для сайта на основе DESIGN.md.
Язык видимого интерфейса: русский.
Ниша: {niche}
Оффер: {offer}
Целевое действие: {primary_cta}
Фото: {photo_roles}
Референсы: {references}
Взять из референсов: {take}
Не копировать: {do_not_copy}
Запреты: generic SaaS, dashboard hero, purple AI, prompt bar, chip cloud, photo-card hero.
Не делай весь сайт сразу. Сначала дай 2-3 визуальных направления с desktop/mobile hero.
```

## Правило Приемки Stitch-Результата

Stitch-результат можно передавать в реализацию только если:

- есть утвержденный `DESIGN.md`;
- выбран один визуальный победитель;
- hero прошел отдельную оценку;
- mobile не ломает оффер и CTA;
- фото не искажает личность;
- все видимые тексты на русском;
- запрещенные паттерны не вернулись;
- понятно, какие экраны и состояния нужно реализовать в Next.js.

Если результата недостаточно, следующий шаг - уточнить `DESIGN.md` или промпт, а не просить Stitch "доделать красиво".
