# Delaem Digital Design Bible v1

## 1. Source-Grounded Design Thesis

**Founder-led ClientFlow Command Center**

Delaem Digital should feel like a premium control room for turning attention into qualified demand. The site is not selling "just websites"; it is presenting a managed ClientFlow System: landing page, intake, AI/qualification readiness, Telegram/email routing, CRM/analytics readiness, attribution, and paid-traffic preparation.

The raw Refero `DESIGN.md` files in `docs/design-references/refero/` are source references, not styles to copy. Their job is to define useful visual behaviors: command discipline, premium atmosphere, conversion clarity, architectural restraint, cinematic authority, mission scale, AI minimalism, dashboard intelligence, luxury trust, and bold technical confidence.

The final identity must be Delaem Digital's own: dark, cinematic, founder-led, architectural, controlled, high-ticket B2B, system-aware, and conversion-oriented. It must not become generic SaaS, crypto/cyber theatre, an info-product landing page, or a replica of any reference.

The hero offer remains fixed:

> "Собираю бизнесу управляемую систему привлечения клиентов"

## 3. Final Visual Principles

- **Dark first:** the product lives on black, graphite, obsidian, and deep slate surfaces.
- **Founder-led:** the site should feel like a serious operator is personally building and managing the system.
- **System over decoration:** every line, panel, route, metric, and signal should imply a real ClientFlow function.
- **Premium restraint:** fewer visual objects, sharper hierarchy, more controlled negative space.
- **Conversion clarity:** CTAs, pricing, forms, and consent states must be unmistakable and calm.
- **One primary accent:** copper/warm premium action is the main conversion signal.
- **No generic SaaS:** avoid default blue/purple gradients, playful blobs, template dashboards, and fluffy section copy.
- **No visual noise:** do not stack cards, charts, glow, glass, and badges just because they look technical.
- **No card spam:** cards are for meaningful units, not every sentence.

## 4. Final Color Direction

The final palette must be Delaem Digital's own. Reference colors are evidence, not source material to copy.

- **Base Black / Graphite:** primary canvas for cinematic depth. Avoid a pure SpaceX black-only identity; use enough graphite variation to create readable layers.
- **Elevated Obsidian:** premium panels, pricing surfaces, and founder/system modules. It should feel polished, not plastic.
- **Deep Slate / Control Surface:** diagnostic areas, route maps, form containers, and secondary panels.
- **Off-White Primary Text:** main readability color; crisp but not sterile.
- **Muted Steel Secondary Text:** descriptions, captions, metadata, subdued labels.
- **Copper / Warm Premium Primary Action:** main CTA and selected/high-value accents. This borrows luxury logic from Slash without copying gold directly.
- **Steel-Blue Secondary Action:** secondary links, route-line interactions, subtle technical affordances.
- **Diagnostic Blue:** analytics, scan, audit, and system-state details only.
- **Success Green:** status only: sent, connected, ready, passed.
- **Danger Red:** validation and failure only.

Forbidden as final identity colors: neon lime, violet primary, generic SaaS blue primary, Calendly light theme, pure black-only severity, and decorative rainbow gradients.

## 5. Typography Rules

The current Manrope / Onest direction remains. Do not introduce paid or unavailable fonts unless separately approved.

- Hero typography should be large, confident, and restrained in weight.
- Display text can be bold, but it must not become loud startup typography.
- Body copy should stay readable, direct, and controlled.
- Technical labels may use system/mono-style treatment, but only for diagnostic labels, route steps, badges, and metadata.
- Letter spacing should be careful and readable; avoid artificial luxury tracking on long Russian text.
- Avoid too many font families, fake terminal styling, and copied D-DIN/SpaceX identity cues.
- Use hierarchy through scale, contrast, spacing, and surface relation before adding decorative type effects.

## 6. Surface System

### Canvas

- **Purpose:** full-page cinematic base.
- **Feel:** black/graphite, quiet, premium, not flat.
- **Radius:** none.
- **Border:** none or invisible section transitions.
- **Shadow/glow:** ambient only, never ornamental blobs.
- **Do not use when:** a contained tool surface is needed.

### Section Surface

- **Purpose:** separate major narrative zones without turning them into cards.
- **Feel:** broad deep-slate or graphite band, subtle change in depth.
- **Radius:** none.
- **Border:** optional hairline top/bottom.
- **Shadow/glow:** minimal.
- **Do not use when:** the content is a repeated item; use a card/panel instead.

### Glass / Elevated Panel

- **Purpose:** premium focus areas: hero system layer, final CTA, selective proof blocks.
- **Feel:** translucent obsidian, controlled frost, inner shadow, soft depth.
- **Radius:** 12-18px only when the panel is a major object.
- **Border:** subtle steel/blue-tinted edge.
- **Shadow/glow:** soft and localized.
- **Do not use when:** many repeated cards appear; glass everywhere weakens the premium effect.

### Diagnostic Panel

- **Purpose:** audit output, ClientFlow route state, technical readiness, source attribution concepts.
- **Feel:** Linear/Fey-like disciplined control surface.
- **Radius:** 6-10px.
- **Border:** fine graphite/steel line.
- **Shadow/glow:** almost none; clarity beats decoration.
- **Do not use when:** content is purely marketing copy.

### Input / Form Surface

- **Purpose:** contact form, fields, consent, final CTA intake.
- **Feel:** readable, stable, conversion-focused.
- **Radius:** 8-12px.
- **Border:** clear idle/focus/error states.
- **Shadow/glow:** focus can use subtle steel-blue/copper; no pulsing.
- **Do not use when:** adding decorative inputs without function.

### Pricing Card Surface

- **Purpose:** offer packaging and purchase confidence.
- **Feel:** obsidian trust, strong hierarchy, simple comparison.
- **Radius:** 10-14px.
- **Border:** restrained edge; recommended option may use copper emphasis.
- **Shadow/glow:** premium depth, not floating SaaS tiles.
- **Do not use when:** the price/offer cannot be scanned quickly.

### Recommended Card Surface

- **Purpose:** one high-intent option or preferred path.
- **Feel:** warmer, more decisive, but still dark and controlled.
- **Radius:** same as pricing cards.
- **Border:** copper/warm accent line or corner signal.
- **Shadow/glow:** subtle warm lift.
- **Do not use when:** more than one item is being marked "special."

## 7. Hero Rules

The hero must keep the approved offer copy:

> "Собираю бизнесу управляемую систему привлечения клиентов"

Hero direction:

- Founder-led: the portrait/founder presence is the human anchor, not a decorative afterthought.
- ClientFlow visible: show a controlled route from first touch to qualified lead through meaningful system signals.
- Cinematic and calm: use depth, contrast, and spacing instead of loud animation or overloaded widgets.
- Strategic: the first screen should imply diagnosis, routing, qualification, and launch readiness.
- Conversion-oriented: primary CTA must be visible early and feel decisive.
- Premium: the system layer should look built, not templated.

Hero must not:

- become a generic SaaS dashboard wall;
- overfill with fake metrics and meaningless charts;
- use literal rockets, space scenes, cyber locks, or crypto-like visuals;
- switch the offer copy;
- bury the CTA below decorative content;
- make the portrait so dominant that the system offer arrives too late.

## 8. Section Rules

### Problem / Loss Map

- Show lost leads, unclear intake, unmanaged follow-up, attribution gaps, and weak traffic readiness as a system problem.
- Use restrained diagnostic framing: symptoms, leak points, consequence.
- Avoid fear-mongering, red-heavy design, and endless pain cards.

### ClientFlow System

- Present the offer as an operating route: traffic source, landing, intake, qualification, notification, storage, analytics readiness.
- Use route bars, step states, compact modules, and system signals.
- Each visual unit must map to a real capability.
- Avoid decorative process diagrams that do not explain how leads move.

### Audit Section

- Make it feel like a controlled diagnostic, not a quiz gimmick.
- Use Linear/Fey discipline: status rows, small metrics, readiness signals, careful spacing.
- Diagnostic blue is allowed here; success green and danger red are status-only.
- Do not overload with finance-style charts or fake analytics dashboards.

### Pricing

- Use Calendly clarity and Slash premium trust.
- Make the recommended path obvious but not shouty.
- Copper can identify the preferred action or high-value tier.
- Keep comparison readable on mobile.
- Avoid bargain-market tone, huge discount styling, and overly friendly SaaS cards.

### Final CTA / Form

- The form should feel like a serious intake console: readable, calm, and easy to complete.
- Preserve existing lead behavior, validation, consent, service intent propagation, and success/error flow.
- Use glass/elevated treatment sparingly to make the final conversion moment feel premium.
- Do not add visual complexity that slows submission.

### Footer / Legal Pages

- Keep legal surfaces quiet, readable, and trustworthy.
- Footer should support trust and navigation, not become another marketing section.
- Legal pages do not need hero drama or decorative command-center modules.
- Avoid redesigning legal content during visual passes unless the task explicitly includes it.

## 9. Mobile Rules

- The first screen must feel premium even when compressed.
- CTA must appear early and remain easy to reach.
- Portrait/founder signal should appear without pushing the offer too far down.
- System signals should compress into a few strong elements, not a long stack of dark boxes.
- Cards must not become an endless scroll of similar panels.
- Forms need comfortable field height, clear focus/error states, and readable consent text.
- Cookie banner should remain compact and not block the whole journey.
- No horizontal overflow, clipped Russian text, or button labels squeezed into awkward lines.

## 10. Motion Rules

- Use soft fade/translate for section entry.
- Route-line drawing is allowed when it explains ClientFlow movement.
- Subtle card lift is allowed for actionable cards.
- Form focus transitions should be calm and functional.
- Avoid aggressive parallax, glow pulsing, constant particles, spinning charts, and decorative motion that implies fake complexity.
- Motion should make the system easier to understand, not more theatrical.

## 11. Forbidden Patterns

- Generic SaaS gradient hero.
- Random blue/violet primary.
- Neon lime identity.
- Crypto/cyber visual cliches.
- Literal rockets, planets, mountains, trading terminals, or security-lock tropes.
- Dashboard overload with fake charts.
- Huge glassmorphism everywhere.
- Too many dark cards with equal weight.
- Light theme takeover.
- Decorative charts with no meaning.
- Overly friendly mass-SaaS tone.
- Info-product landing page patterns.
- Changing the hero offer.
- Changing lead flow during a visual layer.

## 12. Codex Implementation Rules

- Work in layers and keep each visual pass scoped.
- Do not edit backend, Supabase, lead API, notification delivery, quiz runtime, env, package files, or deployment config during visual implementation.
- Preserve lead flow, form behavior, analytics behavior, legal pages, consent links, and runtime isolation.
- Use existing project fonts and design system direction where possible.
- Prefer targeted class/CSS changes over broad rewrites.
- Do not introduce dependencies unless separately approved.
- Do not use reference brands as templates; translate their principles into Delaem Digital's own system.
- Before any implementation, identify the exact sections/files being touched.
- After implementation, run:

```powershell
npm.cmd run build
npx.cmd tsc --noEmit
git diff --check
```
